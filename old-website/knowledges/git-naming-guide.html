Git Naming & Commit Standards Guide

คู่มือมาตรฐานการตั้งชื่อใน Git สำหรับ Branch, Tags, Commits และอื่นๆ

การตั้งชื่อที่ดีและมีมาตรฐานจะช่วยให้ทีมทำงานร่วมกันได้อย่างมีประสิทธิภาพ และสามารถติดตามการเปลี่ยนแปลงของโค้ดได้ง่ายขึ้น คู่มือนี้จะแนะนำมาตรฐานการตั้งชื่อที่นิยมใช้ในอุตสาหกรรม

1. การตั้งชื่อ Branch

โครงสร้างมาตรฐาน

<type>/<scope>-<short-description>
type ที่นิยมใช้:
  • feat – ฟีเจอร์ใหม่
  • fix – แก้บั๊ก
  • refactor – ปรับโครงสร้างโค้ด
  • chore – งานจิปาถะ
  • hotfix – แก้ด่วน
  • test – เพิ่ม/แก้ test
  • docs – เอกสาร
  • perf – ปรับประสิทธิภาพ

scope: เช่น auth, api, dashboard
short-description: สั้น กระชับ ใช้ dash (-) คั่น

ตัวอย่าง Branch Names
  • feat/auth-add-login-otp
  • fix/api-timezone-bug
  • refactor/core-service-layer
  • hotfix/payment-gateway-error
  • docs/update-readme

2. การตั้งชื่อ Tags

ใช้ Semantic Versioning สำหรับการกำหนด version

v<major>.<minor>.<patch>
Standard Versions
  • v1.0.0 – เวอร์ชันแรก
  • v1.1.0 – เพิ่มฟีเจอร์ใหม่
  • v1.1.1 – แก้บั๊ก
  • v2.0.0 – เปลี่ยนแปลงใหญ่
Pre-release Versions
  • v2.0.0-alpha.1 – Alpha release
  • v2.0.0-beta.1 – Beta release
  • v2.0.0-rc.1 – Release Candidate
เมื่อไหร่ควรเพิ่มแต่ละตัวเลข?
  • major: เปลี่ยนแปลง API ที่ทำให้ incompatible
  • minor: เพิ่มฟีเจอร์ใหม่แบบ backward-compatible
  • patch: แก้บั๊กแบบ backward-compatible

3. Commit Message (Conventional Commits)

โครงสร้างพื้นฐาน

<type>(scope): short summary

body (optional)

footer (optional)

รายละเอียด type ที่นิยมใช้

เพิ่มฟีเจอร์ใหม่

ใช้เมื่อเพิ่มความสามารถใหม่ เช่น API ใหม่, UI ใหม่, ฟังก์ชันใหม่

ตัวอย่าง:
feat(auth): add otp verification

Add OTP verification to login process.
Supports SMS and Email delivery.
Closes #221

แก้บั๊ก

เมื่อโค้ดทำงานผิดพลาดและแก้ไขให้ถูกต้อง

ตัวอย่าง:
fix(api): correct timezone conversion issue

Fixed incorrect timezone handling in date fields.
Now properly converts to UTC before saving.
Fixes #342

ปรับโครงสร้างโค้ด

เปลี่ยนโครงสร้างโดยไม่เปลี่ยน behavior ของโปรแกรม

ตัวอย่าง:
refactor(core): simplify user service layer

Extract common validation logic into separate methods.
No functional changes.

งานระบบทั่วไป

Update dependency, config, script ที่ไม่เกี่ยวกับโค้ด

ตัวอย่าง:
chore: update dependencies to latest versions

Updated all npm packages to their latest stable versions.

เอกสาร

README, API documentation, comments

ตัวอย่าง:
docs: add API authentication guide

Added detailed guide on how to use JWT authentication.

  • style – ปรับรูปแบบโค้ด (formatting, spacing, lint)
  • test – เพิ่มหรือแก้ test (unit test, integration test)
  • perf – ปรับประสิทธิภาพ (ลด latency, ลด memory)
  • build – แก้ระบบ build (Dockerfile, webpack, vite)
  • ci – แก้ workflow CI/CD (GitHub Actions, Jenkins)
  • revert – ย้อนการเปลี่ยนแปลง (revert commit)

4. การตั้งชื่อ Pull Request (PR) 

[type]: summary of change
ตัวอย่าง PR Titles
  • [feat]: implement dashboard filters
  • [fix]: resolve worker memory leak
  • [refactor]: restructure authentication module
  • [docs]: update installation guide

5. การตั้งชื่อ Release Branch

รูปแบบที่นิยม:
  • release/2025.03 – ตามปี.เดือน
  • release/v1.4.0 – ตาม version
  • release/sprint-32 – ตาม sprint

6. การตั้งชื่อ Environment Branch

Branch สำหรับ environments ต่างๆ:
  • dev – Development environment
  • staging – Staging environment
  • uat – User Acceptance Testing
  • production – Production environment

Best Practices

  • ใช้ภาษาอังกฤษในการตั้งชื่อและเขียน message
  • เขียน commit message ที่สื่อความหมายชัดเจน
  • แยก commit ตามหน้าที่ ไม่รวมหลายอย่างในครั้งเดียว
  • ใช้ present tense เช่น "add" ไม่ใช่ "added"
  • เขียน summary ไม่เกิน 50 ตัวอักษร
  • เขียน body เพื่ออธิบายรายละเอียด (ถ้าจำเป็น)
  • อ้างอิง issue หรือ ticket number ใน footer
